📱 3. Responsive Print UI — Four-Tier Upgrade

A systematic upgrade of all listing and reporting pages to support mobile card views alongside desktop tables. The strategy: tables get d-none d-md-block d-print-block (hidden on mobile, visible on desktop/print); mobile card wrappers get d-md-none d-print-none (visible on mobile only, hidden in print). A shared CSS component system (.vk-member-card, .vk-card-header, .vk-card-body, .vk-card-row, .vk-card-actions) provides consistent card styling across all pages. A 2-column CSS grid (.vk-cards-wrapper) activates at ≥ 480 px for efficient screen use.

🐛 4. Bug Fix — users.php Card Layout
users.php — Cards Too Narrow / Content Misaligned on Mobile 6th May 2026  •  Branch: feat/responsive-print-ui-tier4  •  Commit: bb93d2e
FIXED
  • Root cause: Bootstrap's default card-body padding (1 rem) stacked with the inner wrapper's p-3 class (another 1 rem), consuming 64 px of horizontal space on each side on narrow mobile screens
  • Second cause: .vk-cards-wrapper applied a 2-column CSS grid, making each card approximately 133 px wide on 375 px phones — far too narrow to display email, last-login date, and 2–3 action buttons
  • Fix applied: Changed card-body → card-body p-0 to eliminate double-padding; removed vk-cards-wrapper class from the card container to prevent the 2-column grid (single-column layout retained for this content-heavy page)

🏗 6. Key Architecture Decisions
Client-Side vs. Server-Side Card Rendering Strategy
  • PHP loop (client-side DataTable): Used for pages where all data is pre-loaded in PHP — customers, loans, transactions, dormant members, budget, death analysis. Card search synced to DataTable via data-search attribute + filterXxxCards(term) function called in drawCallback
  • JS drawCallback (server-side AJAX DataTable): Used for pages where data loads via AJAX — petty_cash, expenses, users, manage_contributions ledger. renderXxxCards(api) rebuilds card HTML from api.rows({page:'current'}).data() after every DataTable draw
  • XSS safety: vkEscU(s) / vkEscL(s) helpers escape all user-supplied strings inserted via JavaScript innerHTML
Financial Ledger — $ledger_rows[] Collection Pattern
  • The financial ledger has highly complex per-row computation (dynamic month columns, entrance fee distribution logic, surplus/deficit calculation) inside a single foreach loop
  • Solution: intermediate $ledger_rows[] array collects computed summary values (compact()) at end of each table row iteration
  • Card view consumes $ledger_rows array in a separate PHP loop below the table, with simplified display (Total Contributed, Balance, Target, Aid) — per-month columns intentionally omitted as unsuitable for card layout
Deployment — HTTPS Webhook vs. FTP/SSH Approaches
  • Why not SSH: cPanel shared hosting blocks inbound SSH from external IPs; GitHub Actions would need special allowlisting
  • Why not FTP: FTP active mode is blocked (most modern hosting); FTP passive mode uses random high ports (1024–65535) for data connections — hosting provider firewall blocks inbound connections on these ports from GitHub Actions IPs, causing partial uploads and data corruption
  • Why HTTPS Webhook: Port 443 is universally open on all shared hosting. Server pulls its own update — no inbound connection required from GitHub. Bearer token authentication. Runs git reset --hard origin/main guaranteeing server matches GitHub exactly
